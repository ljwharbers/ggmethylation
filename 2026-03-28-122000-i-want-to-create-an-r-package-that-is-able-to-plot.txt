‚óŹ Here are three architectural approaches for the package:

  ---
  Approach A ‚ÄĒ Single monolithic function

  One function does everything: reads the BAM, parses MM/ML tags, and produces
  the plot.

  plot_methylation(bam = "sample.bam", region = "chr1:1000-2000",
                   mod_code = "m", group_tag = "HP")

  Pros: Dead simple API. Cons: No way to inspect the parsed data; BAM is re-read
   every time you tweak a plot parameter ‚ÄĒ slow for iteration.

  ---
  Approach B ‚ÄĒ Two-layer: parse then plot (recommended)

  Separate data loading from plotting. Users call read_methylation() once to get
   a structured object, then pass it to the plot function.

  meth <- read_methylation("sample.bam", region = "chr1:1000-2000",
                           mod_code = "m", group_tag = "HP")
  plot_methylation(meth, sort_by = "group", max_reads = 200)

  Pros: BAM is parsed once; users can inspect, filter, or export the data; plots
   can be quickly re-rendered with different parameters. Cons: Slightly more
  verbose API.

  ---
  Approach C ‚ÄĒ Custom ggplot2 geoms

  Implement geom_reads() and geom_methylation() as proper ggplot2 extensions, so
   users build plots layer by layer.

  Pros: Most ggplot2-idiomatic; maximum composability. Cons: Significantly more
  complex to implement; awkward to combine two panels (top + bottom) in the
  ggplot2 paradigm.

  ---
  My recommendation is B. The two-layer design gives you fast iteration (parse
  once, plot many times), lets users inspect the intermediate data for debugging
   or downstream use, and keeps the plotting API clean. It's also much easier to
   test each layer independently.

  Does this approach make sense, or do you want to discuss any of the options?
